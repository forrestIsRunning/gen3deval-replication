# 前端评测上下文与可视化说明

## 当前交互逻辑

前端现在以一个明确的评测上下文为核心：

1. 选择 `Manifest`：决定本轮数据集范围，例如 `manifest_render10.jsonl` 或 `manifest_120.jsonl`。
2. 选择 `资产`：决定要查看或运行 VLM 的具体 3D 资产 `uid`。
3. 选择 `VLM`：决定评分模型，目前只保留 `qwen3-vl-235b-a22b-instruct` 和 `qwen3-vl-235b-a22b-thinking`。

`Run Selected VLM` 只会对当前选中的一个真实资产运行真实 VLM 评测。这样可以让分数、reason、渲染图、几何表都能和 `uid + model` 一一对应，避免把不同模型或不同数据集的结果混在一起。

## 为什么前端去掉 limit

`limit` 适合命令行批处理和调试，但不适合当前前端主流程。

前端的目标是交互式审计：用户选中一个资产和一个 VLM 模型，然后运行并立即查看该资产的多视角证据、VLM 分数、几何指标和失败原因。因此前端不再要求用户输入 `limit`。

批量评测仍然可以通过脚本执行：

```bash
uv run python scripts/score_assets.py \
  --manifest data/processed/manifest_render10.jsonl \
  --model qwen3-vl-235b-a22b-instruct
```

如果需要限制批量数量，命令行仍支持：

```bash
uv run python scripts/score_assets.py \
  --manifest data/processed/manifest_120.jsonl \
  --model qwen3-vl-235b-a22b-instruct \
  --limit 20
```

## 多视角可视化

每个已渲染资产会展示两组图：

- `RGB`：用于观察外观、结构、材质、纹理和文本还原。
- `Normal`：用于观察表面连续性、破面、噪声、过度平滑、几何异常。

VLM 评分也基于这些真实渲染图，不使用 mock。

## 图表如何和模型对应

前端调用 `/api/evaluation?manifest=...&model=...`，后端会按当前 manifest 的 `uid` 集合和当前 VLM 模型过滤：

- `多维平均分`：只统计当前 manifest + 当前 model 的 VLM 分数。
- `几何分布`：只显示当前 manifest 内资产的可计算几何指标，并高亮当前资产。
- `渲染验证`：只显示当前 manifest 内已有渲染记录，并高亮当前资产。
- `样本结果`：只展示当前 manifest + 当前 model 的资产评分卡片。

## 论文想法到当前实现

当前 POC 不是把论文指标全部一次性做完，而是分层落地：

- `Gen3DEval`：落地为多视角 RGB/Normal + VLM rubric，输出多维评分和 reason。
- `T3Bench`：落地为视觉质量和文本对齐分开展示，不只看 overall。
- `GPT-4V Eval3D`：落地为 VLM 作为视觉 judge，并保存自然语言解释。
- `Eval3D`：落地为几何、渲染、VLM、报告分层组合，而不是单一黑盒分数。
- `PointFlow / point cloud metrics`：MMD、Coverage、1-NNA、JSD 暂列为模型级 research eval，适合后续批量生成结果对比。
- `Point-E / Shap-E`：CLIP R-Precision、P-FID、P-IS 暂列为后续低成本代理指标。
- `Textured Mesh Quality Assessment / GET3D / Text2Tex / TEXTure`：落地为纹理材质维度 `texture_material`，并强调几何表不能替代材质判断。
- `ULIP / OpenShape`：作为后续 3D-text/image embedding 对齐指标方向；当前先用 VLM 的 `text_fidelity` 覆盖语义还原。

## 核心 trade-off

- 几何客观指标：用函数直接算，便宜、可复现、适合 scale。
- 语义、视觉、还原度：用 VLM，看多视角 RGB/Normal。
- 报告、解释、错误归因：用 LLM 汇总已有指标和 VLM reason，不让纯 LLM 直接凭空打 3D 分。
